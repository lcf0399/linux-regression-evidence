# mprotect shared-dirty state audit - lab 摘要

这轮补充实验用于确认 `shared_dirty_full_toggle_64m` workload 在
`v6.12.77`、`v6.19.9` 和 `akpm/mm mm-unstable 444fc9435e57` 上是否比较了同一种用户态 mapping 状态。

它不是性能实验，也不是 old-faster 证据。它的目的只是排除类似
`MADV_PAGEOUT` 那种 caveat：不同内核版本实际操作的 page/VMA 状态并不一样。

## 矩阵

| CPU | Guest memory | 每个内核 run 数 |
| ---: | ---: | ---: |
| 1 | 14336 MiB | 5 |
| 2 | 14336 MiB | 5 |
| 4 | 14336 MiB | 5 |
| 8 | 16384 MiB | 5 |
| 16 | 32768 MiB | 5 |

`1/2/4/8 CPU` 三个内核都完成了 5/5。`16 CPU` 这一行里 `v6.19.9`
有一次 QEMU failure，所以 `v6.19.9` 是 4 次成功；成功的 run 中 state-shape
指标仍和其他版本一致。

## 解释

这支持把 mprotect 这条比较看成同一种 shared-dirty 4 KiB PTE workload
状态下的比较。它不能证明 culprit commit，但它说明 mprotect 这条不像
`MADV_PAGEOUT` 后续发现的情况：后者新老版本实际 THP backing 状态不同。

因此，这轮 state audit 支持当前机制解释：剩余 timing gap 更可能来自
`change_pte_range()` shared-dirty PTE 实现路径的额外开销，而不是用户态
mapping 状态不一致。
