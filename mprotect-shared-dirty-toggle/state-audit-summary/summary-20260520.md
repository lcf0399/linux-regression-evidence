# mprotect Shared-dirty State Audit - Lab Summary

This follow-up run checks whether the `shared_dirty_full_toggle_64m`
workload compares the same userspace mapping state across `v6.12.77`,
`v6.19.9`, and `akpm/mm mm-unstable 444fc9435e57`.

It is not a performance run and is not old-faster evidence. It is meant to
rule out a `MADV_PAGEOUT`-style caveat where different kernel versions end
up operating on materially different page or VMA state.

## Interpretation

This supports treating the mprotect comparison as a same-state shared-dirty
4 KiB PTE workload comparison. It does not prove the culprit commit, but it
does make the mprotect result different from the `MADV_PAGEOUT` follow-up,
where later checks found that the compared kernels were not operating on
the same actual THP backing state.

The state audit therefore supports the current mechanism interpretation:
the remaining timing gap is more plausibly in the `change_pte_range()`
shared-dirty PTE implementation path than in a mismatch of userspace mapping
state.
